# 调用图和 Source Insight 风格方案

Neovim 里没有一个完全等同 Source Insight 的事实标准，但有几条可行路径。

## 默认方案：LSP 调用层级

本配置默认提供：

| 操作 | 快捷键 |
| --- | --- |
| 谁调用当前函数 | `<leader>ci` 或 `gai` |
| 当前函数调用谁 | `<leader>co` 或 `gao` |
| 查找引用 | `gr` 或 `<leader>cr` |
| 当前文件大纲 | `<leader>cs` |

优点是稳定、轻量、跟 LazyVim 现有能力一致。缺点是它是“调用列表/调用层级”，不是完整图数据库。

## 可选方案一：litee-calltree.nvim

项目：`ldelossa/litee-calltree.nvim`

特点：

- 基于 LSP call hierarchy。
- 提供可展开/折叠的 incoming/outgoing call tree。
- 相对成熟，GitHub 星标和维护情况比很多小插件好。

本配置已经在 `user-config/lua/plugins/callgraph.lua` 里写好插件块，但默认 `enabled = false`。需要试用时改成 `true`。

## 可选方案二：calltree.nvim

项目：`daishengdong/calltree.nvim`

特点：

- README 明确写的是 Source Insight style relation tree。
- 支持 LSP 和 cscope 两种 parser。
- 作者也列出了一些窗口布局和 LSP 异步上的已知问题。

如果你特别怀念 Source Insight，可以单独实验它，但我不建议默认启用。

## 可选方案三：callgraph.nvim

项目：`barreiroleo/callgraph.nvim`

特点：

- 基于 LSP call hierarchy 导出 Graphviz DOT。
- 更接近“画图”，适合分析局部调用关系。
- 目前项目还年轻，要求 Neovim 0.12+，且 README 里说明 nightly 更推荐。

如果你的目标是生成图片或 DOT 文件，而不是编辑器侧边栏树，可以考虑它。

## 实际建议

先用内置 LSP 调用层级和 Aerial 大纲。等 Go/C++ 项目里的 LSP 跳转稳定后，再打开 `litee-calltree.nvim` 试用。C/C++ 大项目如果需要更 Source Insight 的体验，重点不是插件，而是先保证 `compile_commands.json`、clangd index 和项目根目录识别正确。
